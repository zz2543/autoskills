# APO-SkillsMD 实验框架设计

本文整合文献精读（EvoSkills, GEPA, Promptbreeder, EvoPrompt, SkillsBench）与多轮讨论结论，系统描述 APO-SkillsMD 的方法框架与实验设计。每个组件都从问题本身的结构性质推导，而非从现有论文借鉴拼凑。

---

## 1. 研究问题与核心假设

### 1.1 问题定义

给定任务 $t$ 和初始技能集合 $\mathcal{S}_0 = \{s_1, \dots, s_k\}$（来自技能市场，功能相似但实现各异），目标是演化出一个技能 $s^*$ 使任务完成质量最大化，同时控制部署成本：

$$
s^* = \arg\max_{s} \left[ \text{PassRate}(s, t), \quad -\text{ExecutionCost}(s, t) \right]
$$

**关键约束**：
- 技能 $s$ 是**代码 + 文本的混合体**（`SKILL.md` + `scripts/*.py` + 资源文件）
- 评估必须通过实际沙盒执行，不能依赖 LLM 主观判断
- SkillsBench 提供确定性验证器，每个任务有唯一 ground-truth 输出规范

### 1.2 核心假设（三条）

**假设 1：冗余是资源，不是问题**
技能市场上多个功能相似但实现各异的技能，天然构成理想的进化种群——提供了多样性，而多样性是遗传算法的必需条件。

**假设 2：执行 Trace 是最忠实的变异信号**
技能包含可执行代码，代码的好坏不能靠 LLM 猜测，必须实际执行才能验证。执行轨迹（函数级的成功/失败、异常栈、输出差异）是比 LLM 合成反馈更忠实的 ground-truth 信号。

**假设 3：有效性与简洁性天然冲突**
一个技能可以通过堆砌指令和防御代码提高 pass rate，但会增加 agent 执行的 token 成本。这两个目标之间存在 Pareto 权衡，单目标优化会破坏权衡。

### 1.3 方法架构概览

四个组件直接对应上述假设：

```
假设 1 ──→ ① 冗余池初始化
假设 2 ──→ ② 沙盒执行 + Trace 收集
            ③ Trace-guided 变异 + 模块级交叉
假设 3 ──→ ④ Pareto 多目标选择
```

每个组件都可以用一句话回答"为什么必须要这个"。不是因为前人论文用了，而是因为去掉它会破坏对应假设的实现。

## 2. 四组件框架详细设计

### 2.1 组件①：冗余池初始化

#### 目标

构造初始种群 $P_0$，大小 $N_0 \geq 10$，来源于市场已有技能。

#### 具体流程

```
Step 1: 在技能市场中检索任务 t 相关的 top-K skills
        - 基于任务描述 + skill 的 frontmatter（description、适用场景）做 BM25/embedding 匹配
        - 取 K = 8~12（需消融确定）

Step 2: 过滤明显低质量的 skills
        - 运行一遍安全检查（见第 3 节），拒绝硬违规的 skills
        - 对剩余 skills 运行一次快速 smoke test，淘汰完全不能执行的

Step 3: 如果 K' < N_0，用 LLM 补充生成
        - LLM 从任务描述生成 (N_0 - K') 个 skill，注入多样性
        - 这一步对标 Promptbreeder 的 zero-order 初始化

Step 4: 输出初始种群 P_0
```

#### 关键设计决策

- **不在此阶段做任何评分或排序**——保留原始多样性
- **LLM 补充生成数量 ≤ 30%**——避免稀释冗余池的真实结构信息
- **种群大小不能为 1**——否则后续 Pareto 和 crossover 全部失效

#### 可能的变体（作为 Exp 3 对照）

| 变体名称                    | 初始种群来源                       | 对照目的                   |
| ----------------------- | ---------------------------- | ---------------------- |
| From-Scratch            | LLM 随机生成 N_0 个 skill         | 验证"冗余池是资源"假设           |
| From-Single-Best        | 只用市场评分最高的 1 个 skill，变异产生 N_0 | 验证多样性的价值               |
| From-Lamarckian         | Agent 裸跑收集成功 trace，反推 skill  | 对标 Promptbreeder 的反推算子 |
| **From-Redundant-Pool** | 市场冗余池（本方法）                   | —                      |

### 2.2 组件②：沙盒执行 + Trace 收集

#### 目标

对每个候选技能 $s$ 计算评估向量 $(\text{PassRate}(s), \text{ExecutionCost}(s), \text{Trace}(s))$。

#### 执行架构

```
┌───────────────────────────────────────────┐
│  Docker 沙盒（network=none, 只读文件系统）│
│  ┌─────────────────────────────────────┐  │
│  │  Agent (Claude Code) + skill s     │  │
│  │  ↓                                  │  │
│  │  执行任务 t 的输入                    │  │
│  │  ↓                                  │  │
│  │  产生输出 x                          │  │
│  │  ↓                                  │  │
│  │  SkillsBench 验证器 → pass/fail    │  │
│  └─────────────────────────────────────┘  │
└───────────────────────────────────────────┘
         │
         ↓
  收集三类信号：
  ① PassRate：所有 test case 的通过率
  ② ExecutionCost：agent 执行过程消耗的 token 数
  ③ Trace：函数级执行记录（调用栈、成功/失败行号、异常信息）
```

#### Trace 的结构化表示

每次执行产生一个 Trace 记录：

```python
Trace = {
    "skill_id": str,
    "task_id": str,
    "success": bool,
    "execution_tokens": int,
    "module_events": [
        {
            "module": "scripts/parse.py:tokenize",   # 模块名
            "entered": True,                          # 是否被执行
            "exceptions": [],                         # 抛出的异常
            "duration_ms": int,
            "output_summary": str                     # 前 200 字符
        },
        ...
    ],
    "final_output": {"files_created": [...], "stdout": str}
}
```

#### 关键设计决策

- **ExecutionCost 使用 token 而非代码行数**——技能是用来控制 agent 执行的，真正的成本是 agent 执行时读写的 token
- **Trace 粒度到函数级**——太粗（文件级）丢信息，太细（行级）噪声大
- **每个 skill 在 ≥3 个 test case 上评估**——单 case 易受随机性影响

### 2.3 组件③：Trace-guided 变异 + 模块级交叉

这是方法最复杂的部分，也是相对 EvoSkills/GEPA 的核心差异点。

#### 2.3.1 变异（Mutation）

**输入**：单个技能 $s$ + 其 Trace
**输出**：变异后的技能 $s'$

**流程**：
```
1. 识别 Trace 中的 failure points
   - 哪些模块抛出了异常
   - 哪些 assertion 失败

2. 调用 LLM：给出 failure trace + skill 原文，要求针对性修改
   prompt: "The skill failed at module X with error Y.
            Modify the skill to fix this specific failure.
            Keep unrelated parts unchanged."

3. 返回新技能 s'
```

**关键**：LLM 不做"全局重写"，只做**针对 trace 证据的定向修改**。这保证修改的可追溯性，也让"w/o Trace"消融有意义的对照。

#### 2.3.2 模块级交叉（Crossover / Merge）

**输入**：两个技能 $A$ 和 $B$，以及它们的 Trace
**输出**：融合后的子代 $C$（或 None）

**交叉设计的 5 个关键决策**：

| 决策        | 选择              | 理由                       |
| --------- | --------------- | ------------------------ |
| Parent 数量 | 2               | 模块归属可清晰跟踪；3+ 会让 LLM 决策模糊 |
| Parent 选择 | Pareto 前沿首尾配对   | 最大化互补性（一个偏准 × 一个偏简）      |
| 交叉粒度      | 模块级（函数 + MD 章节） | 字符级会产生语法垃圾；全体重写失去精确控制    |
| 模块选择依据    | Trace 归因        | Trace 是 ground-truth     |
| 结构对齐方式    | LLM 辅助功能槽对齐     | 两个 skill 的函数名不同，按名字对齐会错  |

**完整算法**：

```python
def crossover(A: Skill, B: Skill, trace_A: Trace, trace_B: Trace) -> Skill | None:
    # --- 门禁（借鉴 GEPA 的互补性约束）---
    if not is_pareto_optimal(A) or not is_pareto_optimal(B):
        return None
    if not has_complementary_modifications(A, B):
        return None  # 改的模块完全重合，merge 无意义

    # --- Step 1: 功能槽对齐（LLM 调用 #1，轻量）---
    #   输入：A 和 B 的函数签名 + docstring
    #   输出：功能槽映射 {slot_name: {"A": fn_A, "B": fn_B}}
    slot_map = llm_align_functional_slots(A, B)

    # --- Step 2: 按 Trace 决定每个代码模块的 winner（确定性）---
    winners = {}
    for slot_name, candidates in slot_map.items():
        score_A = success_count(trace_A, candidates["A"]) - fail_count(trace_A, candidates["A"])
        score_B = success_count(trace_B, candidates["B"]) - fail_count(trace_B, candidates["B"])
        winners[slot_name] = "A" if score_A > score_B else "B"

    # --- Step 3: SKILL.md 章节决策（规则驱动）---
    md_decisions = {
        "description":   pick_shorter(A.md.description, B.md.description),
        "workflow":      pick_consistent_with_code(A.md.workflow, B.md.workflow, winners),
        "examples":      merge_union(A.md.examples, B.md.examples),
        "constraints":   merge_union(A.md.constraints, B.md.constraints),
    }

    # --- Step 4: LLM 缝合（LLM 调用 #2，核心）---
    #   输入：A、B、winners、md_decisions、接口不匹配信息
    #   输出：完整 child skill（附带适配器代码）
    child = llm_stitch(A, B, winners, md_decisions)

    # --- Step 5: 可执行性兜底 ---
    if not passes_syntax_check(child):
        child = llm_repair(child, error_msg)
        if not passes_syntax_check(child):
            return None  # 丢弃

    return child
```

#### 2.3.3 逃逸机制（Escape / Diversity Injection）

**触发条件**：连续 K 代（K=3）中 Pareto 前沿无改进，或种群多样性（模块差异度）低于阈值。

**动作**：注入 1~2 个新个体：
- 一半来自 Promptbreeder 式的 zero-order 生成（从任务描述重新生成）
- 一半来自 Lamarckian 反推（从最近的成功 trace 反推 skill）

逃逸不是独立算子，而是变异步骤的兜底分支，避免种群塌缩。
 
### 2.4 组件④：Pareto 多目标选择

#### 目标

从当前种群 + 新生成的候选中，选出 $N_0$ 个个体进入下一代。

#### 两个目标

$$
f_1(s) = \text{PassRate}(s) \quad \text{（最大化）}
$$
$$
f_2(s) = -\text{ExecutionCost}(s) \quad \text{（最大化，即最小化 token 开销）}
$$

#### 算法：NSGA-II 式非支配排序 + 拥挤度截断

```
1. 对合并后的候选集 C（大小 ≈ 2N_0），做非支配排序：
   - Front 1 = 当前 Pareto 前沿
   - Front 2 = 移除 Front 1 后的 Pareto 前沿
   - ...

2. 按 Front 顺序填入下一代种群 P'：
   for front in fronts:
       if len(P') + len(front) ≤ N_0:
           P' ← P' ∪ front
       else:
           # 部分选取：按拥挤度（前沿上邻居的稀疏程度）降序
           remaining = N_0 - len(P')
           P' ← P' ∪ top_k_by_crowding(front, remaining)
           break

3. 返回 P'
```

#### 关键参数

- **种群大小 $N_0 \geq 10$**：太小会让 Pareto 前沿退化成单点
- **保留比例 top_k ≥ 3**：保留前沿多个权衡点供 crossover 使用
- **拥挤度计算**：在 (pass_rate, token_cost) 二维平面上计算邻居距离

## 3. 安全约束

安全不是 Pareto 目标，而是**硬约束**——违规个体直接淘汰，不进入种群。

### 3.1 拦截三层

#### 硬拦截（直接淘汰）

使用 Bandit + 正则静态扫描，检测：

| 类别        | 检测点                                             |
|-----------|-------------------------------------------------|
| 任意代码执行    | `eval()`, `exec()`, `compile()`, `__import__` |
| Shell 注入  | `os.system`, `subprocess(shell=True)`, 反引号    |
| 沙盒逃逸尝试    | `/proc/`, `/sys/`, `ptrace`, `ctypes`          |
| 外部网络      | `requests`, `urllib`, `socket` 到非白名单域名      |
| Prompt 注入 | SKILL.md 中的 "ignore previous instructions" 等 |

#### 软拦截（丢弃但不计为违规）

运行时检测：

| 类别        | 检测方式                                           |
| --------- | ---------------------------------------------- |
| 硬编码任务答案   | skill 内出现与 test expected output 的 hash 匹配的长字面量 |
| 超出职责的文件操作 | 写入 `/app/` 外的路径                                |
| 资源炸弹      | 超时 / 内存超限                                      |

#### 不管

代码风格、依赖 CVE、LLM 废话注释、技能"不够通用"——这些要么由沙盒隔离解决，要么由 Pareto 的 simplicity 目标自然惩罚。

### 3.2 论文里的价值

被过滤的个体**保留在日志里**，可以报告一个数据点：

> "We filter out XX% of evolved candidates that attempt to bypass the task via hardcoding or external API calls, demonstrating that LLM-driven evolution reliably explores reward-hacking strategies absent explicit constraints."

这本身是个有意思的发现，证明安全约束是必需的，不只是工程细节。

## 4. 完整算法流程

把四个组件串起来的主循环：

```
Algorithm: APO-SkillsMD Evolution
Input: Task t, market skill set M, budget B
Output: Best evolved skill s*

1.  P_0 ← InitializeFromRedundantPool(M, t, size=N_0)
2.  for s in P_0:
        s.eval ← SandboxExecute(s, t)   # (pass_rate, tokens, trace)
    # 此时每个 s 都有 Pareto 向量和 trace

3.  for generation = 1 to B do
        # --- 变异 ---
        mutants = []
        for s in P_{gen-1}:
            s' ← TraceGuidedMutation(s, s.trace)
            if passes_safety_check(s'):
                s'.eval ← SandboxExecute(s', t)
                mutants.append(s')

        # --- 交叉 ---
        offspring = []
        front = current_pareto_front(P_{gen-1})
        for (A, B) in complementary_pairs(front):
            C = crossover(A, B, A.trace, B.trace)
            if C is not None and passes_safety_check(C):
                C.eval ← SandboxExecute(C, t)
                offspring.append(C)

        # --- 逃逸（条件触发）---
        if stagnant_for_K_gens(3):
            escape_skills = zero_order_regenerate(t, num=2)
            for e in escape_skills:
                e.eval ← SandboxExecute(e, t)
                offspring.append(e)

        # --- Pareto 选择 ---
        C_all = P_{gen-1} ∪ mutants ∪ offspring
        P_{gen} = NSGA_II_Select(C_all, size=N_0)

4.  s* ← arg max_s in P_B [ pass_rate(s) ]   # 或按用户指定权衡
5.  return s*
```

## 5. 实验设计

四个实验，每个回答一个具体问题，之间有递进关系：

### 5.1 Exp 1：整体性能对比

**问题**：APO-SkillsMD 进化出的 skills 能否超越现有方法？

| Baseline                | 说明                     | 来源          |
| ----------------------- | ---------------------- | ----------- |
| No-Skill                | Agent 裸跑               | 自跑          |
| Self-Generated          | LLM 一次性生成 skill，不优化    | 自跑          |
| Market-Best             | 市场上评分最高的通用 skill       | 市场          |
| Human-Expert            | SkillsBench 官方专家 skill | SkillsBench |
| EvoSkills (复现或引用)       | 当前 SOTA                | 论文 / 自跑     |
| **APO-SkillsMD (Ours)** | 本方法                    | 自跑          |

**指标**：
- 主指标：Pass Rate（在 SkillsBench 全 87 任务上）
- 辅助指标：Execution Cost（token 消耗）

**预期结果**：
- 超过 Self-Generated 和 Market-Best：高置信
- 接近或超过 Human-Expert（53.5%）：中置信
- 超过 EvoSkills（71.1%）：低置信，需要谨慎评估
- **Token Cost 显著低于 EvoSkills**：高置信（Pareto 对齐了简洁性）

### 5.2 Exp 2：组件消融

**问题**：四个组件各自贡献多大？

| 变体             | 改动                                | 验证什么                |
|----------------|----------------------------------|---------------------|
| Full           | 完整方法                               | —                   |
| w/o Trace      | 降级为 GEPA 式谱系归因（纯模块变化检测）            | Trace 归因相对谱系归因的增量价值  |
| w/o Crossover  | 只有单点变异，不做跨技能交叉                    | 模块级交叉的价值            |
| w/o Pareto     | 贪心只选 pass rate top-N             | 多目标选择的价值            |
| w/o 逃逸         | 种群塌缩时不注入新个体                       | 多样性维护的价值            |

**设计亮点**：4 个消融精确对应 4 个组件，不多不少。

**关键对比 "w/o Trace"**：
- 这不是简单地"不用 trace"，而是**用 GEPA 的谱系归因替代 trace 归因**作为 merge 时的决策依据
- 如果 Full 显著优于 w/o Trace，直接证明**"trace 比谱系更有价值"**这一核心论点
- 这是区分我们和 GEPA 的最强证据

### 5.3 Exp 3：初始化策略对比

**问题**：从冗余技能池出发是否比其他初始化方式更好？

| 初始种群           | 说明                                 |
|----------------|------------------------------------|
| From-Scratch   | LLM 随机生成 N_0 个 skill              |
| From-Single-Best | 市场最高评分 1 个 skill，变异至 N_0       |
| From-Lamarckian | Agent 裸跑成功 trace 反推 skill         |
| **From-Redundant-Pool** | 市场冗余池（本方法）                   |

**固定**：其他所有组件保持一致（Trace 变异 + 模块交叉 + Pareto）。

**关键对比 From-Lamarckian**：对标 Promptbreeder 的反推算子。如果比 From-Redundant-Pool 差，说明冗余池中的**多样性**比"从成功 trace 反推"更有价值——强化我们的核心 insight。

### 5.4 Exp 4：进化过程分析（定性）

**问题**：进化过程中发生了什么？技能是怎么从通用变特异的？

**分析内容**：
- 进化曲线（pass rate vs 代数）
- Pareto 前沿随代数的移动轨迹
- 进化前后 skill 内容的 diff 分析（LLM 总结关键变化）
- 按 SkillsBench 的 11 个领域分组，看哪些领域进化增益最大
- **General→Specific 轨迹可视化**：用 skill 内容的嵌入距离衡量"任务特异性"的演化

**这个实验不追求数字提升**，而是提供定性理解，让审稿人"看到"进化过程。

最后一条（General→Specific 可视化）直接承担摘要里 "task-specificity as the inherent direction" 这句话的举证责任——必须有一张图证明这个轨迹是真实的。

## 6. 和现有工作的边界

### 6.1 和 EvoSkills 的关系：互补而非竞争

- EvoSkills 回答："给定一个任务，如何用 LLM 自我进化出一个 skill（在 GT 反馈缺失时）？"
- APO-SkillsMD 回答："给定一个已有冗余 skill 市场，如何用进化算法挖掘和组合其中的价值？"

论文里的定位：

> We do not view our work as a direct competitor to EvoSkills but as a complementary paradigm. EvoSkills optimizes a single skill under opaque reward; APO-SkillsMD exploits the structure of existing skill markets through population-level evolution. These paradigms could be integrated in future work (e.g., EvoSkills-style surrogate verification driving our mutation step).

### 6.2 和 GEPA 的关系：降级复杂度，升级信号质量

- GEPA 在**纯文本 prompt** 上提出了 merge + Pareto 的思路
- APO-SkillsMD 将这套思路**迁移到代码+文本混合体**，并用 **trace 替代 GEPA 的谱系归因**作为模块选择依据
- GEPA 是我们最直接的方法论祖先；w/o Trace 消融精确对标 GEPA 的决策方式

### 6.3 和 Promptbreeder 的关系：借鉴消融结论，不借鉴 9 算子

- Promptbreeder 自己的消融证明 "初始化主导 >> 各变异算子"——直接支持我们"冗余池初始化"的设计价值
- 我们不使用 Promptbreeder 的 9 算子堆砌，但借用其 zero-order 和 Lamarckian 反推作为**对比初始化**和**逃逸机制**

## 7. 一体性检验：从动机到实验的闭环

这份设计的一体性可以通过"每个设计决策是否对应一个明确的假设/问题/差异化"来检验：

| 设计决策          | 对应假设/差异化                        | 实验中由哪个验证                 |
|---------------|---------------------------------|---------------------------|
| 冗余池初始化        | 假设 1（冗余是资源）                     | Exp 3（From-Redundant-Pool vs 其他）|
| 沙盒执行 + Trace  | 假设 2（trace 是最忠实信号）              | Exp 2 的 w/o Trace 对比 GEPA 谱系 |
| Trace-guided 变异 | 假设 2                            | Exp 2 的 w/o Trace            |
| 模块级交叉         | 相对 EvoSkills 的差异化（种群 + crossover） | Exp 2 的 w/o Crossover        |
| 功能槽对齐         | 相对 GEPA 的差异化（结构可变）              | 实现细节（无独立消融）               |
| Pareto 多目标    | 假设 3（效性-简洁性冲突）                  | Exp 2 的 w/o Pareto           |
| 逃逸机制          | 种群多样性保护                         | Exp 2 的 w/o 逃逸               |
| 从冗余池到任务特异化    | Key finding（task-specificity 是路径）| Exp 4 的 General→Specific 可视化 |

每一行都贯通：假设 → 设计 → 实验 → 论文结论。没有"为什么要这个"回答不出来的组件。

### 8.3 风险与退路

| 风险                                       | 退路                                            |
|------------------------------------------|-----------------------------------------------|
| 跑不过 EvoSkills 的 71.1%                    | 摘要降调为 "competitive with"，强调 token cost 优势    |
| 冗余池在 SkillsBench 上找不到足够的相关技能             | 用 LLM 合成冗余池（多样性从 sampling 得到），但改称 "synthetic redundant pool" |
| Trace 归因在函数级粒度下噪声过大                     | 降级到 module/file 级                             |
| 交叉产生的 child 大部分失败（接口适配难）                | 提高 LLM repair 轮数，或简化 slot_map 到只对齐大模块         |
| Task-specific emergence 在可视化上不明显         | Key finding 降调为 "distributional shift"，不强调单向轨迹 |

每个风险都有明确的降级方案，避免实验不顺时陷入僵局。
